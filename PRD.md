## 1. 产品概述

### 1.1 背景
同学们在日常饮食中常常面临"今天吃什么"的选择困难。本系统通过记录用户的饮食历史，利用余弦相似度算法，为用户推荐口味属性相似的食谱，帮助用户发现新的美食选择。

### 1.2 目标
构建一个轻量级食谱推荐系统，实现食谱管理、用户管理、饮食记录与智能推荐四大核心功能。

### 1.3 成功指标
- 支持至少 50 条食谱的管理
- 推荐响应时间 < 2 秒
- 支持 10+ 用户同时使用

## 3. 详细功能描述

### 3.1 食谱管理

#### 3.1.1 食谱数据结构

```
Recipe {
    id:          int (自增主键)
    name:        string (食谱名称，唯一)
    calories:    int (热量，单位 kcal)
    created_at:  datetime
    updated_at:  datetime
}
```

#### 3.1.2 属性（口味标签）数据结构

采用 **EAV (Entity-Attribute-Value)** 设计模式，支持动态扩展属性：

```
Attribute {
    id:          int (自增主键)
    name:        string (属性名称，唯一，如"甜"、"辣"、"酸"、"苦")
}

RecipeAttribute {
    id:          int (自增主键)
    recipe_id:   int (外键 → Recipe.id)
    attribute_id: int (外键 → Attribute.id)
    value:       int (属性值，0-10，表示程度)
}
```

**设计说明**：
- 新增口味属性时，只需在 `Attribute` 表插入一条记录
- 每个食谱对每个属性有一个 0-10 的分值，表示该口味的强度
- 例如：麻婆豆腐 → 辣:9, 麻:8, 辣:7（"麻"可作为新增属性）

> **可选简化方案**：如果口味只做二值化（有/无），则将 `value` 改为 `boolean`，或改用 `SET` 类型。推荐用 int 分值保留表达能力。

#### 3.1.3 CRUD 操作

| 操作 | 描述 |
|------|------|
| **添加食谱** | 输入名称、热量、各口味分值 → 写入 Recipe + RecipeAttribute |
| **修改食谱** | 更新名称/热量/口味分值 |
| **删除食谱** | 删除 Recipe 及其关联的 RecipeAttribute、EatingRecord |
| **列出食谱** | 分页展示，支持筛选（名称模糊搜索、热量区间、口味过滤） |

### 3.2 用户系统

#### 3.2.1 用户数据结构

```
User {
    id:            int (自增主键)
    username:      string (唯一)
    password_hash: string (哈希存储，不存明文)
    created_at:    datetime
}
```

#### 3.2.2 功能

| 操作 | 描述 |
|------|------|
| **注册** | 输入用户名 + 密码（密码需加密存储，如 bcrypt/sha256+盐） |
| **登录** | 验证用户名 + 密码，创建会话（session 或 JWT token） |
| **登出** | 清除会话 |
| **鉴权中间件** | 除注册登录外，其他 API 均需登录态 |

### 3.3 饮食记录

#### 3.3.1 数据结构

```
EatingRecord {
    id:          int (自增主键)
    user_id:     int (外键 → User.id)
    recipe_id:   int (外键 → Recipe.id)
    eaten_at:    datetime (吃的日期时间)
}
```

#### 3.3.2 功能
- 用户选择食谱并记录"吃过"
- 同一食谱可多次记录
- 查看个人饮食历史（按时间倒序）

### 3.4 推荐系统

#### 3.4.1 算法：余弦相似度

**核心思路**：
1. 将每个食谱表示为一个**属性向量**，向量维度 = 所有 Attribute 的数量
2. 以用户吃过的所有食谱的属性向量**平均值**作为"用户口味向量"
3. 计算用户口味向量与所有**未吃过**食谱向量的余弦相似度
4. 返回相似度 Top-N 的食谱

**公式**：

```
similarity(A, B) = (A · B) / (||A|| × ||B||)
```

其中 A 为用户平均口味向量，B 为候选食谱属性向量。

#### 3.4.2 冷启动策略

- 新用户尚未有任何饮食记录时：
  - 推荐**全局吃过次数最多的食谱**（热门推荐）
  - 或随机推荐若干食谱

#### 3.4.3 推荐结果

- 返回 Top-5 或 Top-10 推荐列表
- 推荐列表排除用户已吃过的食谱
- 附带相似度分数（供调试/展示用）

## 4. 技术方案建议

### 4.1 技术栈选择（建议，可按需调整）

| 层次 | 技术选项 | 考虑 |
|------|----------|------|
| **后端语言** | Python (Flask/FastAPI) 或 Go | Python 生态好，适合快速原型；Go 性能更优 |
| **数据库** | SQLite（本地单机）或 MySQL（多用户） | MVP 建议 SQLite，部署简单 |
| **前端** | CLI 终端 / Web (HTML+JS) / Tkinter | MVP 建议 CLI 或简单 Web，减少前端工作量 |
| **密码存储** | bcrypt / hashlib+盐 | 不可逆加密 |

### 4.2 架构图（简化）

```
┌─────────────┐     ┌──────────────┐     ┌──────────┐
│   CLI / Web  │────▶│ 业务逻辑层    │────▶│ Database │
│  (前端展示)  │     │ (推荐算法等)   │     │ (SQLite) │
└─────────────┘     └──────────────┘     └──────────┘
```

### 4.3 数据库 ER 图（文本描述）

```
User ──1:N──▶ EatingRecord ──N:1──▶ Recipe ──N:N──▶ Attribute
                                        │               │
                                        └─── RecipeAttribute ───┘
                                              (关联表，含分值)
```

## 5. 非功能需求

| 需求 | 说明 |
|------|------|
| **数据持久化** | 所有数据存入数据库，程序重启不丢失 |
| **密码安全** | 密码不可逆加密存储 |
| **输入校验** | 防止 SQL 注入、XSS（如果 Web） |
| **错误处理** | 用户操作失败时有明确提示 |
| **可扩展性** | 新增属性无需改表结构，不改代码逻辑 |

## 8. 风险与缓解措施

| 风险 | 缓解 |
|------|------|
| 推荐质量不高（冷启动） | 用热门推荐兜底 |
| 数据库设计过于灵活导致查询复杂 | EAV 模式下统计查询可用 JOIN + GROUP BY |
| 余弦相似度对稀疏向量不敏感 | 确保新用户先吃几道菜再推荐 |
