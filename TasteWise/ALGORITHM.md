# TasteWise 推荐算法说明

本文档说明 TasteWise 当前推荐系统的完整算法流程，覆盖五味向量建模、加权欧氏距离、余弦相似度、Top-K 排序、筛选条件、推荐理由和地域口味扩展。

## 1. 算法目标

TasteWise 的推荐目标是：根据用户对酸、甜、苦、辣、咸五种味道的偏好，从食堂菜品数据中选出最匹配的 Top-K 菜品，并给出匹配分数和推荐理由。

算法输入包括：

- 用户五味偏好：`[acid, sweet, bitter, spicy, salty]`
- 菜品五味画像：每道菜同样包含五个味道评分
- 可选权重：表示用户更重视哪一种味道
- 可选筛选条件：食堂、价格区间、已互动菜品排除
- 可选地域口味：用于“老家是哪里”模块

算法输出包括：

- 推荐菜品列表
- `match_score` 匹配分数
- `reasons` 推荐理由

## 2. 五味向量建模

用户和菜品都被表示为五维向量：

```text
[酸, 甜, 苦, 辣, 咸]
```

每一维取值范围为 `1` 到 `5`：

- `1` 表示该味道较弱
- `5` 表示该味道较强

示例：

```text
用户偏好: [3, 2, 1, 5, 4]
菜品画像: [2, 1, 1, 5, 4]
```

## 3. 加权欧氏距离

加权欧氏距离用于衡量用户五味偏好与菜品五味画像之间的绝对差距。

公式：

```text
distance = sqrt(sum(weights * (user - dish)^2))
max_distance = sqrt(sum(weights * 4^2))
match_score = (1 - distance / max_distance) * 100
```

说明：

- 每一维最大差值为 `4`，因为评分范围是 `1` 到 `5`
- 距离越小，匹配度越高
- 权重越大，该味道对最终结果影响越明显
- 输出会限制在 `0` 到 `100`

适用场景：

- 用户明确知道自己每种味道想要多少
- 需要支持“辣味更重要”“甜味不重要”等权重调整

## 4. 余弦相似度

余弦相似度用于衡量用户口味向量和菜品口味向量的方向是否一致。

公式：

```text
similarity = dot(user, dish) / (norm(user) * norm(dish))
match_score = similarity * 100
```

说明：

- 更关注整体口味趋势是否一致
- 对具体数值差异没有加权欧氏距离敏感
- 当前实现中余弦相似度不使用自定义权重
- 输出会限制在 `0` 到 `100`

适用场景：

- 用户更关注整体口味类型
- 菜品强度不同，但味道方向相似时仍希望获得较高分

## 5. 筛选与排序流程

推荐函数 `recommend_dishes()` 的主要流程如下：

```text
校验用户五味向量
    ↓
校验菜品字段
    ↓
按食堂、价格、已互动菜品进行筛选
    ↓
为每一道候选菜品计算匹配分数
    ↓
生成推荐理由
    ↓
按 match_score 降序排序
    ↓
返回 Top-K 推荐结果
```

排序规则：

```text
match_score 降序
dish_id 升序
```

当多个菜品分数相同时，使用 `dish_id` 保持结果稳定。

## 6. 推荐理由生成

系统会比较用户画像与菜品画像，在五味维度上找出匹配度较高的味道，生成推荐理由。

示例：

```text
辣味匹配度较高
咸味匹配度较高
酸味匹配度较高
```

如果用户选择了“老家是哪里”，推荐理由中还会补充地域口味说明。

## 7. 地域口味扩展

地域口味模块不是独立替代五味算法，而是在五味算法基础上的增强。

当前支持的地域画像包括：

- 川渝地区
- 湖南 / 江西
- 江浙沪
- 广东 / 福建
- 东北地区
- 西北 / 新疆
- 华北 / 山东
- 云贵地区

系统会将地域口味画像与用户五味偏好融合，并对命中地域关键词的菜品给予小幅加分。

示例：

```text
用户偏好: [3, 3, 1, 3, 3]
川渝画像: [2, 1.5, 1, 5, 4.5]
融合后画像更偏辣、偏咸
```

## 8. 异常与边界处理

当前算法包含以下校验：

- 用户五味向量必须正好包含 5 个值
- 五味评分必须在 `1` 到 `5` 之间
- 权重必须正好包含 5 个正数
- 菜品数据必须包含必要字段
- `top_k` 必须大于 0
- 无候选菜品时返回空推荐结果
- 不支持的算法名称会抛出明确错误
