## 核心原则

1. **先同步，再干活** — 每次编程前必须先同步最新代码
2. **先 PR，再合并** — 所有变更必须通过 Pull Request，禁止直接推送到 master
3. **先 Review，再合并** — PR 创建后提醒用户 Review，用户确认后方可合并
4. **新手友好** — 解释操作时通俗易懂，避免纯术语堆砌

## 编程前准备

### 每次开始编程前，必须执行以下步骤：

```bash
# 1. 切换到 master 分支
git checkout master

# 2. 拉取最新代码
git pull origin master

# 3. 检查是否有未提交的本地修改
git status

# 4. 如果本地有未提交的修改，询问用户如何处理
#    （丢弃 / 暂存 / 提交）
```

> ⚠️ **重要**：如果在执行 `git pull` 时遇到网络错误（如 SSL 连接失败），应尝试：
> - `git -c http.sslBackend=openssl pull origin master`
> - 或使用 `gh` CLI 通过 API 获取最新代码
> - 如果所有网络方式都失败，**必须告知用户**并协商处理方案

### 确认清单

- [ ] 本地 master 与远程 master 一致
- [ ] 没有未提交的本地修改（或已和用户确认处理方式）
- [ ] 已了解当前要开发的功能范围和 PRD 要求

## 开发流程

```
1️⃣ 同步最新代码
    │
2️⃣ 创建功能分支
    │
3️⃣ 编写代码
    │
4️⃣ 本地测试（如果项目有测试）
    │
5️⃣ 提交到分支
    │
6️⃣ 推送到远程
    │
7️⃣ 创建 Pull Request
    │
8️⃣ 提示用户 Review → 用户确认后合并
```

## 代码提交规范

### 禁止行为

| 行为 | 说明 | 正确做法 |
|------|------|---------|
| ❌ 直接推送到 master | 绕过 Review 流程 | 创建分支 → PR → 合并 |
| ❌ 强制推送（force push）到共享分支 | 覆盖他人提交 | 除非用户明确要求且知晓风险 |
| ❌ 提交未完成的代码 | 破坏项目稳定性 | 完成功能或修复后再提交 |
| ❌ 提交含敏感信息的代码 | API 密钥、密码泄露 | 使用环境变量或配置文件 |

### 允许行为

| 行为 | 说明 |
|------|------|
| ✅ 从 master 创建新分支 | 标准开发流程 |
| ✅ 向自己的分支推送代码 | 随时可以推送 |
| ✅ 创建 PR 请求合并 | 标准协作流程 |
| ✅ 在 PR Review 通过后合并 | 标准协作流程 |

## 分支命名规范

```
<类型>/<简短描述>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/user-login` |
| `fix/` | 修复 bug | `fix/readme-path-error` |
| `docs/` | 文档变更 | `docs/update-readme` |
| `refactor/` | 重构代码 | `refactor/recommend-engine` |
| `test/` | 添加测试 | `test/add-recommender-tests` |

**规则**：
- 全部小写
- 用连字符 `-` 分隔单词
- 描述简洁明了

## Commit 规范

### 格式

```
<类型>: <简短描述>

<可选的详细说明>
```

### 类型

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档 |
| `refactor` | 重构 |
| `test` | 测试 |
| `chore` | 杂项（构建、依赖等） |

### 示例

```bash
feat: 添加用户注册功能
fix: 修复推荐算法除零错误
docs: 更新 README 运行指引
refactor: 提取推荐算法为独立模块
```

### 规范

- 描述用中文或英文均可，但要前后一致
- 简短描述不超过 50 字
- 复杂变更在正文中说明原因

## PR 规范

### 创建 PR 时

```bash
gh pr create \
  --base master \
  --head <分支名> \
  --title "<类型>: <描述>" \
  --body "## 变更内容

<详细的变更说明>

## 测试

- [ ] 本地测试通过
- [ ] 功能验证正常

## 关联

- Closes #<issue编号>（如有关联 Issue）"
```

### PR 必须包含

- [ ] **标题** — 简洁说明变更内容
- [ ] **正文** — 描述了改了什么、为什么改
- [ ] **关联 Issue**（如果有）— 方便追踪

### 创建 PR 后

**必须提示用户**：
> "PR #<编号> 已创建，请前往 Review 确认后合并"
> "https://github.com/Bistu-OSSDT-2026/Preference/pull/<编号>"

### 处理冲突

如果 PR 出现合并冲突：
1. 先向用户解释冲突原因（谁改了同一个文件的哪部分）
2. 提出解决方案供用户选择
3. 在用户同意后执行解决操作

## 文件操作规范

### 创建文件

- 新文件必须放在合理的目录结构中
- 文件名用小写字母加连字符（如 `data_manager.py`）
- Markdown 文件首尾用 `---` 分隔线

### 修改文件

- 修改前先读取文件内容
- 避免大面积重写已存在的文件（除非用户要求）
- 修改后验证文件完整性

### 删除文件

- 必须告知用户并确认
- 确认没有其他文件依赖被删除的文件

---

## 安全规范

- **绝不** 将密码、API 密钥、Token 硬编码到代码中
- 敏感信息应通过环境变量或配置文件传入
- git 仓库提交前检查是否误入了敏感信息

---

## 沟通规范

### 对用户说话时

- **通俗易懂**：避免堆砌术语，用类比和比喻解释技术概念
- **分步骤**：复杂操作拆成小步骤，每步都有预期结果
- **主动汇报**：每个操作完成后主动告知结果（成功/失败/异常）
- **出现错误时**：说明错误原因 + 建议解决方案

### 对用户的问题

- 先理解需求，再确认理解是否正确
- 有多个方案时列出选项及优缺点，让用户选择
- 用户是初学者时，主动解释"为什么"而不只是"怎么做"
